# FluxPlayer AI 开发守则

> 本守则基于项目实际开发经验制定，所有使用 AI 辅助开发的场景均须遵循。

---

## 一、头文件与依赖管理

- **最小化头文件暴露**：头文件只暴露接口所需的最少依赖，实现细节用前向声明或 pImpl 隔离。
- **禁止在头文件中 include 平台 SDK**：AudioToolbox / WinMM / ALSA / CUDA 等平台相关头文件只能出现在 `.cpp` 中。
- **类型替换**：头文件中避免使用 `GLuint`、`AVFrame *` 等第三方类型，改用 `unsigned int`、`void *` 或前向声明。
- **统一预编译头**：公共重型头文件（STL、第三方库）统一放入 `include/FluxPlayer/pch.h`，不在各文件重复 include。

## 二、构建系统规范

- **双构建文件同步**：修改 `CMakeLists.txt` 时必须同步修改 `xmake.lua`，反之亦然。
- **生成器表达式**：CMake 中 Debug/Release 差异配置使用生成器表达式，不用 `if(CMAKE_BUILD_TYPE)`。
- **跨编译器写法**：编译选项使用 `target_compile_options` + 编译器检测，不硬编码 GCC/Clang 专属 flag。
- **依赖隔离**：有特殊编译需求的第三方库（如 tinyfiledialogs）单独建 target，不污染主 target 的 PCH。
- **构建文件注释**：每个 target、每个关键选项必须有中文注释说明用途。

## 三、代码注释规范

- **注释语言**本项目统一使用**中文注释**。
- **必须注释的位置**：
  - 每个类/结构体的职责说明
  - 每个公开函数的参数、返回值、副作用
  - 非显而易见的算法逻辑（如 PTS 回绕检测、指数退避）
  - 平台差异分支（`#ifdef _WIN32` 等）
  - 线程安全约束（哪些成员只能在特定线程访问）
- **注释质量**：注释说明"为什么"，而非重复代码字面意思。

```cpp
// 错误示例：
i++;  // i 加 1

// 正确示例：
// PTS 回绕：MPEG-TS 33bit 时间戳约 26.5 小时后归零，检测到跳变超过阈值则重校准基准
if (pts_diff < -kPtsWrapThreshold) { ... }
```

## 四、代码鲁棒性要求

- **资源管理**：所有资源（文件句柄、FFmpeg 对象、GL 纹理）使用 RAII 封装，禁止裸 `new`/`delete`。
- **错误处理**：FFmpeg / OpenGL / 平台 API 的返回值必须检查，错误通过日志系统记录，不静默忽略。
- **线程安全**：跨线程共享数据必须加锁或使用原子操作，注释标明锁的粒度和持有时机。
- **边界检查**：队列操作、缓冲区读写必须检查越界，网络流场景尤其注意 PTS 异常值。
- **降级策略**：硬件加速、高级特性失败时必须有软件降级路径，不能直接崩溃。
- **断言使用**：内部不变量用 `assert`，对外接口用显式错误返回，不混用。

## 五、代码优雅性要求

- **单一职责**：每个类只做一件事。`Player` 负责调度，`GLRenderer` 负责渲染，`AudioOutput` 负责音频，不交叉。
- **pImpl 隔离**：平台相关实现用 pImpl 模式封装，头文件保持平台无关。
- **零拷贝优先**：数据路径上避免不必要的内存拷贝（参考 NV12 零拷贝渲染设计）。
- **命名规范**：类名 `PascalCase`，成员变量 `snake_case_`（尾下划线），常量 `kCamelCase`。
- **函数长度**：单函数不超过 80 行，超过则拆分为私有辅助函数。
- **Magic Number**：所有魔法数字提取为具名常量，注释说明来源。

## 六、文档同步要求

**每次完成功能开发或重构后，必须同步更新以下文档：**

| 文档 | 更新内容 |
|------|---------|
| `README.md` | 功能特性列表、技术栈、跨平台支持状态 |
| `docs/代码剖析.md` | 涉及模块的架构说明、数据流、关键实现细节 |
| `fluxplayer.ini.sample` | 新增/修改/删除的配置项，并附详细中文注释 |
| 相关 `docs/*.md` | 专项技术文档（硬件加速、字幕、网络流等） |

**配置文件规则**：`fluxplayer.ini.sample` 是用户配置的唯一参考模板，每个配置项必须有中文注释说明其作用、取值范围和默认值。新增配置项时注释格式：

```ini
# 说明：<功能描述>
# 取值：<可选值或范围>
# 默认：<默认值>
key = value
```

文档更新与代码修改**在同一个 commit 中提交**，不允许"后续补文档"。

## 七、Commit 规范

遵循 Conventional Commits：

```
<type>(<scope>): <中文描述>

<详细说明，分点列出变更内容>
```

| type | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `refactor` | 重构（不改变行为） |
| `docs` | 仅文档变更 |
| `build` | 构建系统变更 |
| `perf` | 性能优化 |

## 八、AI 辅助开发专项规则

1. **不生成无用代码**：AI 生成的代码必须直接服务于需求，禁止生成"以防万用"的冗余逻辑。
2. **不引入新依赖**：未经确认不得引入项目现有技术栈之外的第三方库。
3. **不破坏现有接口**：重构时保持公开接口稳定，内部实现可自由调整。
4. **生成后必须验证**：AI 生成的构建配置、平台适配代码，必须在目标平台实际编译验证后才能提交。
5. **注释不可省略**：AI 生成代码时注释要求与手写代码完全一致，不因"AI 生成"而降低标准。
6. **文档必须同步**：AI 完成功能后，文档同步更新是交付的一部分，不是可选项。
7. **配置热加载**：新增配置项时，必须在 `Config::load()` 中补充热加载逻辑，确保界面切换时配置能实时生效。
