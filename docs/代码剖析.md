### 第四步：解码线程工作流程 (decodingThread)

**代码位置**：`src/core/Player.cpp:612-1040`

这是播放器的**核心循环**，在后台线程中不断运行。

> **重要**：音频和视频的解码是在**同一个线程中串行完成**的，不是并行解码。
> 实际的执行顺序是：优先接收已解码的帧 → 都接收不到时才去读新的packet → 按stream_index分发给对应解码器。
> 因为解码一帧只需要几毫秒，而播放一帧需要几十毫秒，所以串行完全来得及。

```cpp
void Player::decodingThread() {
    AVPacket* packet = av_packet_alloc();
    Frame rawFrame;

    while (!shouldQuit_) {
        bool frameReceived = false;

        // ====== 第1步：优先尝试接收已解码的视频帧 ======
        // FFmpeg的解码是异步的：一个packet可能解出多个frame
        // 所以必须先把解码器缓冲区里的帧全部取出来
        if (videoDecoder_->receiveFrame(rawFrame)) {
            // 从环形队列获取可写槽（队列满时阻塞等待渲染线程消费）
            // 对标 ffplay frame_queue_peek_writable，实现生产者/消费者背压
            Frame* writable = videoQueue_->peekWritable();
            if (!writable) break;  // abort 信号（stop/quit）
            videoDecoder_->prepareFrame(rawFrame.getAVFrame(), *writable);
            videoQueue_->push();  // 推进写索引，notify 渲染线程
            frameReceived = true;
        }

        // ====== 第2步：尝试接收已解码的音频帧（同一个线程，串行） ======
        if (audioDecoder_->receiveFrame(rawFrame)) {
            Frame* audioWritable = audioQueue_->peekWritable();
            if (!audioWritable) break;
            audioDecoder_->convertToS16(rawFrame.getAVFrame(), *audioWritable);
            audioQueue_->push();
            frameReceived = true;
        }

        // ====== 第3步：如果没接收到帧，读取新的压缩数据包 ======
        if (!frameReceived) {
            if (!demuxer_->readPacket(packet)) {
                break;  // 文件读完了
            }

            // 根据stream_index判断是视频包还是音频包，分发给对应解码器
            if (packet->stream_index == demuxer_->getVideoStreamIndex()) {
                videoDecoder_->sendPacket(packet);   // 送入视频解码器
            }
            else if (packet->stream_index == demuxer_->getAudioStreamIndex()) {
                audioDecoder_->sendPacket(packet);   // 送入音频解码器
            }

            av_packet_unref(packet);
        }
    }
}
```

**这个线程做什么？**
1. 优先从解码器接收已解码好的帧（视频帧和音频帧串行接收）
2. 如果没有帧可接收，才去读取新的压缩数据包
3. 根据数据包的 `stream_index` 分发给视频或音频解码器
4. 解码后的帧通过 `FrameQueue` 背压机制传递给渲染/音频线程

**为什么是串行而不是并行？**
- 解码速度远快于播放速度（解码1帧只要几毫秒，播放1帧要~40毫秒）
- `FrameQueue` 固定容量（本地 4 视频 / 10 音频，网络 8 / 20），队列满时解码线程阻塞等待，自动节流到渲染速度
- 串行设计更简单，避免了多线程解码的复杂同步问题

**FrameQueue 环形队列设计（对标 ffplay）**：
- 预分配固定大小的 Frame 数组，消除每帧 `make_shared` + `av_frame_alloc` 开销
- `peekWritable()` + `push()`：生产者侧，队列满时 `condition_variable` 挂起
- `peek()` + `next()`：消费者侧，`next()` 唤醒生产者
- keep-last 机制：视频队列保留最后渲染帧，暂停/截图时可复用
- `abort()` + `flush()`：stop/seek 时干净唤醒阻塞线程

**数据变化过程（对应代码中的具体函数调用）**：

```
第3步 readPacket:
  Demuxer::readPacket(packet)          → 得到 AVPacket（压缩码流）
  ↓                                       代码: av_read_frame(m_formatCtx, packet)
  ↓
第3步 sendPacket:
  VideoDecoder::sendPacket(packet)     → 将 AVPacket 送入FFmpeg解码器内部缓冲区
  ↓                                       代码: avcodec_send_packet(m_codecCtx, packet)
  ↓
第1步 receiveFrame:
  VideoDecoder::receiveFrame(rawFrame) → 从解码器取出 AVFrame（原始像素数据）
  ↓                                       代码: avcodec_receive_frame(m_codecCtx, frame.getAVFrame())
  ↓                                       此时 rawFrame 内部的 AVFrame 包含解码后的YUV数据
  ↓                                       同时计算 PTS: pts = avFrame->pts * av_q2d(m_timeBase)
  ↓
第1步 prepareFrame:
  VideoDecoder::prepareFrame(...)      → 将 AVFrame 准备为可渲染格式的 Frame
                                          硬件解码: av_hwframe_transfer_data() → NV12 → av_frame_ref()
                                          软件解码: av_frame_ref()（YUV420P零拷贝）或 sws_scale()
                                          最终得到的 Frame 通过 videoQueue_->push() 入队
```

> **注意 sendPacket 和 receiveFrame 的异步关系**：
> FFmpeg的解码是异步的——`sendPacket()` 只是把数据送进去，不会立即产出帧。
> 解码器内部可能需要积累多个packet才能输出一帧（比如B帧需要参考前后帧），
> 也可能一个packet解出多个帧。所以代码中是**先尝试receiveFrame，再去readPacket**，
> 确保解码器缓冲区里的帧都被取出来。

### 第五步：视频渲染循环 (Player::run)

**代码位置**：`src/core/Player.cpp:219-310`

主线程负责渲染视频帧，这是一个**精确的定时循环**：

```cpp
void Player::run() {
    double lastFramePTS = 0.0;
    
    while (!shouldQuit_ && !window_->shouldClose()) {
        // 1. 处理窗口事件（键盘、鼠���）
        glfwPollEvents();
        
        // 2. 从视频队列取出一帧
        auto frame = popVideoFrame();
        if (!frame) {
            // 队列空，等待解码线程填充
            std::this_thread::sleep_for(std::chrono::milliseconds(10));
            continue;
        }
        
        // 3. 计算这一帧应该延迟多久显示（音视频同步）
        double framePTS = frame->getPTS();
        double delay = avSync_->computeFrameDelay(framePTS, lastFramePTS);
        
        // 4. 如果延迟为负，说明这帧太晚了，应该丢弃
        if (delay < 0) {
            droppedFrames_++;
            continue;  // 跳过这一帧
        }
        
        // 5. 等待到正确的显示时间
        std::this_thread::sleep_for(
            std::chrono::duration<double>(delay)
        );
        
        // 6. 渲染这一帧到屏幕
        renderer_->renderFrame(
            frame->getData()[0],  // Y平面
            frame->getData()[1],  // U平面
            frame->getData()[2],  // V平面
            frame->getLinesize()[0],
            frame->getLinesize()[1],
            frame->getLinesize()[2]
        );
        
        // 7. 更新视频时钟
        avSync_->updateVideoClock(framePTS);
        lastFramePTS = framePTS;
        
        // 8. 交换缓冲区，显示到屏幕
        window_->swapBuffers();
        
        // 9. 渲染UI（进度条、控制面板等）
        if (renderCallback_) {
            renderCallback_();
        }
    }
}
```

**关键点**：
- 这是一个**帧率控制循环**，每次显示一帧
- `computeFrameDelay()` 是音视频同步的核心，计算每帧应该等待多久
- 如果视频落后音频太多，会**丢帧**追赶进度

### 第六步：音频播放回调 (audioOutputCallback)

**代码位置**：`src/core/Player.cpp:782-900`

音频输出是**回调驱动**的，系统需要音频数据时会调用这个函数：

```cpp
size_t Player::audioOutputCallback(uint8_t* buffer, size_t bufferSize) {
    size_t bytesWritten = 0;
    
    while (bytesWritten < bufferSize) {
        // 1. 从音频队列取出一帧
        auto frame = popAudioFrame();
        if (!frame) {
            // 队列空，填充静音
            memset(buffer + bytesWritten, 0, bufferSize - bytesWritten);
            break;
        }
        
        // 2. 获取音频数据
        uint8_t* audioData = frame->getData()[0];
        int dataSize = frame->getNbSamples() * audioChannels_ * 2;  // S16格式
        
        // 3. 应用音量
        int16_t* samples = (int16_t*)audioData;
        float vol = muted_ ? 0.0f : volume_.load();
        for (int i = 0; i < frame->getNbSamples() * audioChannels_; i++) {
            samples[i] = (int16_t)(samples[i] * vol);
        }
        
        // 4. 拷贝到输出缓冲区
        size_t bytesToCopy = std::min(dataSize, bufferSize - bytesWritten);
        memcpy(buffer + bytesWritten, audioData, bytesToCopy);
        bytesWritten += bytesToCopy;
        
        // 5. 更新音频时钟
        avSync_->updateAudioClock(frame->getPTS());
    }
    
    return bytesWritten;
}
```

**这个回调做什么？**
1. 从音频队列取出PCM数据
2. 应用音量控制
3. 填充到系统音频缓冲区
4. 更新音频时钟（用于音视频同步）

### 2. VideoDecoder（视频解码器）

**代码位置**：`include/FluxPlayer/decoder/VideoDecoder.h`

**作用**：将压缩的H.264/H.265视频解码为可渲染帧，支持硬件加速

```cpp
class VideoDecoder {
    AVCodecContext* m_codecCtx;     // 解码器上下文
    SwsContext* m_swsCtx;           // 格式转换器（回退路径用）
    AVBufferRef* m_hwDeviceCtx;     // 硬件设备上下文（null=软件解码）
    AVFrame* m_hwTransferFrame;     // 可复用的GPU→CPU传输帧
    
    bool init(AVCodecParameters* codecParams, AVRational timeBase);
    bool sendPacket(AVPacket* packet);      // 送入压缩包
    bool receiveFrame(Frame& frame);        // 接收解码帧
    bool prepareFrame(AVFrame* src, Frame& dst);  // 帧准备（替代原convertToYUV420P）
    bool isHWAccelActive() const;           // 是否正在使用硬件加速
    AVHWDeviceType getHWDeviceType() const; // 供 Player 检测后端类型（CUDA/D3D11VA等）
};
```

**工作流程**：
1. `init()`：查找解码器，尝试硬件加速初始化（按平台优先级）
2. `sendPacket()`：将压缩数据包送入解码器
3. `receiveFrame()`：从解码器取出解码后的帧
4. `prepareFrame()`：硬件帧→GPU→CPU传输(NV12)→零拷贝入队；软件帧→YUV420P零拷贝

**硬件加速初始化**（`initHWAccel()`）：
- macOS：尝试 VideoToolbox
- Windows：按优先级尝试 CUDA(NVDEC) → D3D11VA → DXVA2
- 第一个成功的即用，全部失败则自动回退软件解码
- 新增 `getHWFormat` 回调，确保 FFmpeg 正确协商硬件像素格式

**硬件解码自动降级**（`reinitAsSoftware()`）：

VideoToolbox/CUDA 等硬件解码器遇到参考帧丢失（如视频文件本身有丢帧）时，整个解码线程池进入不可恢复状态，`avcodec_flush_buffers()` 无法恢复。

降级机制：
- `sendPacket()` 内部维护连续失败计数 `m_hwFailCount`
- 连续失败 ≥ 3 次时调用 `reinitAsSoftware()`，完全重建解码器上下文（禁用硬件加速，单线程）
- 重建后立即重发当前 packet，软件解码器对损坏帧会尽力输出花屏而非拒绝
- `init()` 时将 `codecParams` 深拷贝到 `m_savedCodecParams`，供降级重建使用

**CUDA 绿屏修复**（Windows CUDA 后端）：

CUDA + NVIDIA 驱动下 `GL_RG8` 纹理存在兼容性问题，UV 数据采样异常导致绿色画面。解决方案：
- `GLRenderer` 新增 UV 解交错模式：将交错 UV 平面拆分为独立 U/V 平面，复用 YUV420P shader 路径
- `Player` 通过 `getHWDeviceType() == AV_HWDEVICE_TYPE_CUDA` 检测后端，自动启用解交错模式

**FFmpeg API调用**：
- `av_hwdevice_ctx_create()` - 创建硬件设备
- `avcodec_send_packet()` - 送入数据
- `avcodec_receive_frame()` - 接收帧
- `av_hwframe_transfer_data()` - GPU→CPU帧传输
- `sws_scale()` - 格式转换（回退路径）

**实现位置**：`src/decoder/VideoDecoder.cpp`

### 3. AudioDecoder（音频解码器）

**代码位置**：`include/FluxPlayer/decoder/AudioDecoder.h`

**作用**：将压缩的AAC/MP3音频解码为S16 PCM格式

```cpp
class AudioDecoder {
```

### 4. AVSync（音视频同步）

**代码位置**：`include/FluxPlayer/core/AVSync.h`

**作用**：确保音频和视频同步播放，这是播放器最核心的算法

```cpp
class AVSync {
    ClockType clockType_;           // 同步模式
    std::atomic<double> audioClock_;  // 音频时钟
    std::atomic<double> videoClock_;  // 视频时钟
    
    // 核心方法
    double computeFrameDelay(double framePTS, double lastFramePTS);
    bool shouldDropFrame(double framePTS, double threshold);
};
```

**同步策略**：

FluxPlayer使用**外部时钟（系统时钟）**作为基准：

```cpp
// 计算帧延迟的核心算法
double AVSync::computeFrameDelay(double framePTS, double lastFramePTS) {
    // 1. 计算帧间隔（理论延迟）
    double delay = framePTS - lastFramePTS;
    
    // 2. 获取主时钟（系统时钟）
    double masterClock = getMasterClock();
    
    // 3. 计算视频与主时钟的差值
    double diff = framePTS - masterClock;
    
    // 4. 根据差值调整延迟
    if (diff <= -syncThreshold) {
        // 视频落后，减少延迟（加快）
        delay = std::max(0.0, delay + diff * 0.5);
    } else if (diff >= syncThreshold) {
        // 视频超前，增加延迟（减慢）
        delay = delay + diff * 0.5;
    }
    
    return delay;
}
```

**三种时钟模式**：
- `AUDIO_CLOCK`：以音频为基准，视频跟随音频
- `VIDEO_CLOCK`：以视频为基准，音频跟随视频
- `EXTERNAL_CLOCK`：以系统时钟为基准（FluxPlayer使用这种）

**实现位置**：`src/core/AVSync.cpp:74-129`

### 4.1 播放速度控制

**代码位置**：`include/FluxPlayer/core/Player.h`，`src/core/AVSync.cpp`

**作用**：支持 0.5x / 0.75x / 1.0x / 1.25x / 1.5x / 2.0x 六档变速播放，视频和音频同步加速/减速。

#### 时钟驱动机制

变速的核心是让外部时钟以 `rate` 倍实时速率推进。视频渲染循环的帧显示条件为 `nextPTS <= masterClock + 0.005`，主时钟加速后视频帧被更快消耗：

```cpp
// AVSync::getExternalClock()
double elapsed = duration_cast<double>(now - externalClockBase_).count();
return externalClockOffset_ + elapsed * playbackRate_;  // 乘以速率
```

**速率切换防时钟跳变**：切换速率前先锁定当前时钟值，重置基准，避免 `elapsed × newRate` 导致时钟大幅跳变（如 2x→0.5x 时时钟跳退导致视频冻结）：

```cpp
// AVSync::setPlaybackRate(rate)
double current = getExternalClock();
externalClockBase_ = steady_clock::now();
externalClockOffset_.store(current);
playbackRate_.store(rate);
```

#### 快放丢帧（>1.0x）

外部时钟加速后视频帧被快速消耗，对已取出的帧做智能丢帧减少解码压力：
- I 帧永不丢弃（解码基准）
- B 帧按概率丢弃：`(rate-1)/rate`（2.0x 时 50%）
- P 帧仅 2.0x 时以 25% 概率丢弃

**代码位置**：`src/core/Player.cpp` `shouldDropFrameForSpeed()`

#### 音频变速重采样

每次音频回调消耗 `bufferSize × rate` 字节内容，最近邻重采样到 `bufferSize` 字节输出：

```
快放(2x)：消耗 2× 内容 → 压缩到 1× 输出 → 音调升高
慢放(0.5x)：消耗 0.5× 内容 → 拉伸到 1× 输出 → 音调降低
```

音频时钟同步更新：`currentAudioPTS = prevClock + deviceDuration × rate`

**代码位置**：`src/core/Player.cpp` `audioOutputCallback()`

### 5. GLRenderer（OpenGL渲染器）

**代码位置**：`include/FluxPlayer/renderer/GLRenderer.h`

**作用**：使用OpenGL将视频帧渲染到屏幕，支持YUV420P和NV12两种格式

```cpp
class GLRenderer {
    GLuint m_textureY;   // Y平面纹理（亮度，两种格式共用）
    GLuint m_textureU;   // U平面纹理（色度Cb，YUV420P用）
    GLuint m_textureV;   // V平面纹理（色度Cr，YUV420P用）
    GLuint m_textureUV;  // UV交错纹理（GL_RG8，NV12硬件解码零拷贝用）
    
    void renderFrame(uint8_t* yData, uint8_t* uData, uint8_t* vData,
                     int yPitch, int uPitch, int vPitch,
                     bool isNV12 = false,
                     int colorSpace = 0, int fullRange = 0);
};
```

**渲染流程**：

1. **上传YUV/NV12数据到GPU纹理**：

YUV420P 路径（软件解码）：
```cpp
// Y平面（全分辨率，GL_RED 单通道）
glTexSubImage2D(..., GL_RED, GL_UNSIGNED_BYTE, yData);
// U平面（1/2分辨率，GL_RED 单通道）
glTexSubImage2D(..., GL_RED, GL_UNSIGNED_BYTE, uData);
// V平面（1/2分辨率，GL_RED 单通道）
glTexSubImage2D(..., GL_RED, GL_UNSIGNED_BYTE, vData);
```

NV12 路径（硬件解码零拷贝）：
```cpp
// Y平面（全分辨率，GL_RED 单通道，与YUV420P相同）
glTexSubImage2D(..., GL_RED, GL_UNSIGNED_BYTE, yData);
// UV交错平面（1/2分辨率，GL_RG 双通道，U和V交错存储）
glTexSubImage2D(..., GL_RG, GL_UNSIGNED_BYTE, uvData);
```

2. **在片段着色器中转换YUV/NV12到RGB**：

**着色器代码位置**：`assets/shaders/video.frag`

```glsl
uniform int isNV12;      // 0 = YUV420P, 1 = NV12
uniform int colorSpace;  // 0 = BT.601(SD), 1 = BT.709(HD), 2 = BT.2020(UHD)
uniform int fullRange;   // 0 = TV/limited range, 1 = PC/full range

float y = texture(texY, TexCoord).r;
float u, v;

if (isNV12 == 1) {
    // NV12：从 GL_RG8 纹理的两个通道分别读取 U 和 V
    vec2 uv = texture(texU, TexCoord).rg;
    u = uv.r - 0.5;
    v = uv.g - 0.5;
} else {
    // YUV420P：U 和 V 各自独立的单通道纹理
    u = texture(texU, TexCoord).r - 0.5;
    v = texture(texV, TexCoord).r - 0.5;
}

// TV range 量化范围缩放（ITU-R 标准 studio swing）
if (fullRange == 0) {
    y = (y - 16.0/255.0) * (255.0/219.0);
    u = u * (255.0/224.0);
    v = v * (255.0/224.0);
}

// YUV→RGB 转换（根据 colorSpace 选择 BT.601/709/2020 系数）
// 以 BT.709 为例：Kr=0.2126, Kb=0.0722
float r = y + 1.5748 * v;
float g = y - 0.1873 * u - 0.4681 * v;
float b = y + 1.8556 * u;
FragColor = vec4(clamp(r,0,1), clamp(g,0,1), clamp(b,0,1), 1.0);
```

Player 在渲染前从 `AVFrame` 提取色彩元数据传递给着色器：
- `AVFrame::colorspace` → 映射为 colorSpace（0/1/2）
- `AVFrame::color_range` → 映射为 fullRange（0/1）
- 元数据为 `AVCOL_SPC_UNSPECIFIED` 时按分辨率启发式选择（≥720p → BT.709）

**为什么用YUV而不是RGB？**
- 视频编码器输出的就是YUV格式
- YUV可以色度抽样（4:2:0），节省带宽
- GPU可以高效处理YUV到RGB的转换

**实现位置**：`src/renderer/GLRenderer.cpp`

**代码位置**：`include/FluxPlayer/audio/AudioOutput.h`

**作用**：跨平台音频输出，将PCM数据送到扬声器

```cpp
class AudioOutput {
    AudioCallback callback_;  // 音频数据回调

    void resume();  // 恢复
```

**pImpl 重构**：`AudioOutput.h` 使用前向声明 + pImpl 模式，将三平台 SDK（AudioToolbox/WinMM/ALSA）的头文件隔离到 `.cpp` 中，避免污染全局头文件依赖链。

**平台实现**：
- **macOS**：使用AudioToolbox的AudioQueue
- **Windows**：使用WinMM的waveOut
- **Linux**：使用ALSA

**工作原理**：

音频输出是**回调驱动**的：

```
系统音频线程 → 需要数据 → 调用callback_ → Player填充数据 → 播放
```

**macOS实现**（`src/audio/AudioOutput.cpp`）：
```cpp
// AudioQueue回调函数
void AudioOutput::audioQueueCallback(void* userData, 
                                     AudioQueueRef queue,
                                     AudioQueueBufferRef buffer) {
    AudioOutput* output = (AudioOutput*)userData;
    
    // 调用Player的回调获取音频数据
    size_t bytesRead = output->callback_(
        buffer->mAudioData,
        buffer->mAudioDataBytesCapacity
    );
    
    buffer->mAudioDataByteSize = bytesRead;
    
    // 将缓冲区重新入队
    AudioQueueEnqueueBuffer(queue, buffer, 0, nullptr);
}
```

## 线程模型

FluxPlayer使用**三线程模型**：

```
┌──────────────────────────────────────────┐
│           主线程 (Main Thread)            │
│  - 窗口事件处理 (glfwPollEvents)         │
│  - 视频帧渲染 (GLRenderer::renderFrame)  │
│  - UI绘制 (ImGui)                        │
│  - 帧率控制 (sleep)                      │
└──────────────────────────────────────────┘

┌──────────────────────────────────────────┐
│        解码线程 (Decoding Thread)         │
│  - 读取数据包 (demuxer->readPacket)      │
│  - 视频解码 (videoDecoder->receiveFrame) │
│  - 音频解码 (audioDecoder->receiveFrame) │
│  - 填充队列 (pushVideoFrame/pushAudioFrame)│
└──────────────────────────────────────────┘

┌──────────────────────────────────────────┐
│      音频线程 (Audio Thread)              │
│  - 系统音频回调驱动                       │
│  - 从音频队列取数据                       │
│  - 应用音量控制                           │
│  - 更新音频时钟                           │
└──────────────────────────────────────────┘
```

**线程同步**：
- 视频队列和音频队列使用`std::mutex`保护
- 使用`std::atomic`保存状态标志（`shouldQuit_`等）
- Seek 请求使用 `seekMutex_` 保护的 `SeekRequest` 结构体，消除 `seekRequested_`/`seekTarget_` 两个 atomic 之间的 TOCTOU 竞态
- 音视频同步通过时钟（`AVSync`）协调

## Config 模块解析

### 概述

`Config` 是全局配置管理单例，负责配置文件的读写、热重载，以及平台标准目录的管理。

### 平台标准数据目录

配置文件和用户数据（截图、录制）统一存放在平台标准缓存目录下，避免依赖工作目录，且不会被云同步占用空间：

| 平台 | 路径 |
|------|------|
| macOS | `~/Library/Caches/FluxPlayer/` |
| Windows | `%LOCALAPPDATA%\FluxPlayer\` |
| Linux | `~/.cache/FluxPlayer/` |

### 核心方法（`src/utils/Config.cpp`）

| 方法 | 职责 |
|------|------|
| `getAppDataDir()` | 获取平台标准缓存目录，Windows 用 `SHGetFolderPathA(CSIDL_LOCAL_APPDATA)`，macOS 用 `~/Library/Caches`，Linux 遵循 XDG 规范用 `~/.cache` |
| `Config()` | 构造函数，创建数据目录，初始化 `configPath_`、`screenshotDir`、`recordDir` 为绝对路径 |
| `load()` | 从 INI 文件加载配置，不存在则自动生成默认配置，加载后同步日志级别 |
| `save()` | 将当前配置回写到 INI 文件 |
| `checkAndReload()` | 检测文件修改时间，变化时自动重载（界面切换时调用，实现热重载） |

### 目录结构

```
~/Library/Caches/FluxPlayer/   （macOS 示例）
├── fluxplayer.ini      # 配置文件
├── fluxplayer.log      # 日志文件（logFileEnabled=true 时生成）
├── Screenshot/         # 截图保存目录
└── Record/             # 录制文件保存目录
```

### 下游使用

- `Player.cpp`：从 `cfg.recordDir` 读取录制目录，拼接时间戳文件名
- `Screenshot.cpp`：从 `cfg.screenshotDir` 读取截图目录，自动创建子目录
- `Recorder.cpp`：接收完整输出路径，自动创建父目录
- 下游代码无需关心平台差异，Config 构造时已将默认值设为绝对路径

## 字幕模块解析

### 整体数据流

```
Demuxer (AVPacket, stream_index=字幕流)
    ↓
Player::decodingThread() 路由
    ↓
SubtitleDecoder::decode(packet) → vector<Item>{text, startPTS, endPTS}
    ↓
SubtitleManager::addEntry()  ← 解码线程写入（加锁）
    ↓
Controller::renderSubtitles() 每帧查询 getCurrentText(masterClock)
    ↓
ImGui 半透明浮层渲染（底部居中）
```

### SubtitleDecoder（`include/FluxPlayer/subtitle/SubtitleDecoder.h`）

- 封装 FFmpeg `avcodec_decode_subtitle2` API
- 支持文本类字幕：SRT / ASS / SSA / WebVTT / mov_text
- 图形字幕（PGS / DVB-SUB）阶段一跳过，记录 DEBUG 日志
- 用 RAII scope guard 确保 `avsubtitle_free` 在所有路径被调用
- PTS 换算优先级：`end_display_time` > `packet->duration` > 5 秒兜底
- `stripASSOverrides()` 清理 `{\\xxx}` 控制标签，兼容 FFmpeg 4.x 的 `Dialogue:` 前缀格式

### SubtitleManager（`include/FluxPlayer/subtitle/SubtitleManager.h`）

- **零 FFmpeg 依赖**，可被 UI 层安全 include
- 线程安全：单 `std::mutex` 保护 `std::deque<Entry>`
- `addEntry()`：常见路径 O(1) 尾插；乱序时 `upper_bound` 插入；超 500 条从头裁剪
- `getCurrentText()`：正序扫描，同时惰性清除 `endPTS < currentPTS - 5s` 的过期条目

### Controller 字幕渲染（`src/ui/Controller.cpp`）

- `renderSubtitles()` 在 `render()` 最前调用，**独立于 UI 可见性**（UI 自动隐藏时字幕仍显示）
- `loadSubtitleFont()` 在 ImGui 初始化时调用，按平台探测 CJK 字体，用 MergeMode 叠加多字符范围
- `Controller.h` 中 `ImFont*` 用 `void*` 存储，保持头文件零 ImGui 依赖

### Seek 与循环播放

- Seek 时：`SubtitleDecoder::flush()` + `SubtitleManager::clear()`，避免残留旧字幕
- 循环播放重启时：`decodingFinished_` 必须重置为 false，否则渲染循环立即退出

### 配置项（`[Subtitle]` 节）

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| `subtitleEnabled` | 字幕总开关（重开媒体生效） | `true` |
| `subtitleFontScale` | 字幕字体缩放（0.5 ~ 4.0） | `1.4` |
| `subtitleFontPath` | 自定义字体路径（空则自动探测） | 空 |

## 网页视频播放模块

### 整体数据流

```
用户输入网页 URL
    │
    ▼ StreamExtractor::needsExtraction()
    │
    ├─ false → 原有流程（直链/RTSP/RTMP）
    │
    └─ true  → StreamExtractor::extract()
                    │ 调用 yt-dlp -j
                    ▼
               ExtractedStream { videoUrl, audioUrl, headers, isDash, duration }
                    │
                    ├─ isDash=false → Demuxer::open(videoUrl, headers)
                    │
                    └─ isDash=true  → DashMerger::start(videoUrl, audioUrl, headers)
                                           │ FFmpeg API 合并，写入 pipe
                                           ▼
                                      Demuxer::open("pipe:N", "", duration)
```

### StreamExtractor（`src/utils/StreamExtractor.cpp`）

**职责：** 调用 yt-dlp 提取网页视频流信息，解析 JSON 输出。

**关键方法：**
- `needsExtraction(url)`：检测 URL 是否需要提取。直链（含媒体扩展名）、RTSP/RTMP 返回 false；已知平台域名或无扩展名的 HTTP URL 返回 true。
- `extract(pageUrl, formatId, out, error)`：同步调用 yt-dlp，解析 JSON，填充 `ExtractedStream`。应在后台线程调用（阻塞约 2-5 秒）。
- `detectDefaultBrowser()`：通过 `plutil` 读取 macOS plist 检测默认浏览器，映射到 yt-dlp 浏览器名（chrome/safari/firefox/edge）。

**yt-dlp 路径：** 编译时通过 `YTDLP_BUNDLED_PATH` 宏注入（CMakeLists.txt / xmake.lua），优先使用 `third_party/yt-dlp/` 打包版本，找不到则回退到系统 PATH。

**格式选择：** 默认优先 H.264（`bestvideo[vcodec^=avc1]+bestaudio`），避免 AV1/HEVC 在 MKV pipe 中的兼容性问题。

**DASH 判断：** JSON 中存在 `requested_formats` 数组即为 DASH 分离流，第一个对象为视频，第二个为音频。

### DashMerger（`src/utils/DashMerger.cpp`）

**职责：** 在后台线程用 FFmpeg API 将视频流和音频流合并为 MKV，通过 `pipe()` 管道输出。

**关键设计：**
- 用 `pipe(2)` 创建匿名管道，写端由合并线程写入，读端以 `"pipe:N"` 形式传给 `Demuxer::open()`
- 合并时清除 `codec_tag = 0`，避免 mp4a 与 MKV 容器不兼容
- 按 DTS 顺序交错写出（`av_compare_ts`），确保音视频时间戳连续，避免 `Starting new cluster` 警告
- Windows 使用命名管道（`\\.\pipe\fluxplayer_dash_PID`）

**线程安全：** `start()`/`stop()` 应在同一线程调用；`running_` 为 `atomic<bool>`，合并线程通过它感知停止信号。

### Downloader（`src/utils/Downloader.cpp`）

**职责：** 后台线程调用 yt-dlp 下载视频，解析进度/速度，通过回调通知 UI。

**进度解析：** 逐行读取 yt-dlp stderr（`--newline` 参数），匹配 `[download] XX.X% ... at X.XXMiB/s ETA XX:XX` 格式，提取百分比、速度、剩余时间。

**回调：**
- `onProgress(float progress, string speed, string eta)`：每收到进度行触发
- `onFinish(bool ok, string path, string error)`：下载完成或失败时触发

### Demuxer 扩展（`src/decoder/Demuxer.cpp`）

新增重载 `open(url, httpHeaders, knownDuration)`：
- `httpHeaders`：注入 `Referer`、`User-Agent` 等防盗链 headers（`av_dict_set(&opts, "headers", ...)`）
- `knownDuration`：pipe 流无法从容器头读取 duration，手动注入 `m_formatCtx->duration = knownDuration * AV_TIME_BASE`，避免被误判为直播流（否则 seek 和倍速会被禁用）

### Player 扩展（`src/core/Player.cpp`）

**`open()` 新增流程：**
1. `StreamExtractor::needsExtraction()` 检测
2. 若需要：`setState(EXTRACTING)` → `StreamExtractor::extract()` → 根据 `isDash` 决定走 DashMerger 还是直接 URL
3. 保存 `lastPageUrl_`（供画质切换和下载使用）

**`switchQuality(formatId, seekTime)`：**
1. 暂停 → 重新提取指定 formatId 的流 → 停止旧解码线程 → 重置 Demuxer/DashMerger → 打开新流 → 重启解码线程 → seek 到原位置

### Controller 扩展（`src/ui/Controller.cpp`）

**画质按钮（`renderQualityButton`）：** 仅当 `currentQualityLabel_` 非空时显示（即当前播放内容来自网页 URL）。点击弹出画质列表，选择后调用 `player_.switchQuality()`。

**下载按钮（`renderDownloadButton`）：** 仅当 `currentPageUrl_` 非空时显示。点击弹出系统文件夹选择对话框，确认后启动 `Downloader`，底部显示蓝→紫渐变进度条。

**`currentPageUrl_` 同步：** 每帧 `render()` 开始时从 `player_.getLastPageUrl()` 读取，自动跟随播放内容变化。

### 配置项（`[Stream]` 节）

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| `cookiesBrowser` | Cookie 来源浏览器（auto/chrome/safari/firefox/edge/off） | `auto` |
| `cookiesFile` | cookies.txt 文件路径（cookiesBrowser=off 时生效） | 空 |
