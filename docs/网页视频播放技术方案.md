# FluxPlayer 网页视频播放技术方案

## 目标

用户输入任意网页 URL（B站、YouTube、抖音、优酷、爱奇艺等），FluxPlayer 自动识别、提取并播放其中的视频，体验与输入直链无异。

---

## 现状分析

### 已支持
- 直链视频文件（MP4/MKV/AVI 等 HTTP 直链）
- HLS 直播流（`.m3u8`）
- RTSP / RTMP 直播流
- FFmpeg 内置的所有网络协议

### 不支持
- 需要 JavaScript 渲染的网页（B站、YouTube 等）
- 需要登录鉴权的平台
- DASH 分离流（视频/音频分离，需合并）
- 带防盗链 headers 的流（需携带 Referer/Cookie）

### 核心约束
- 无 avfilter（CMakeLists.txt 未链接），无法用 FFmpeg 滤镜管线合并流
- URL 在整个链路中原样传递：`HomeScreen → main → Player::open → Demuxer::open → avformat_open_input`
- 无插件机制，扩展需直接修改源码

---

## 模块设计

### 1. StreamExtractor（新增）

**文件：**
- `include/FluxPlayer/utils/StreamExtractor.h`
- `src/utils/StreamExtractor.cpp`

```cpp
struct StreamInfo {
    std::string title;
    std::string videoUrl;
    std::string audioUrl;   // 空 = 单一流或无音频
    std::string headers;    // "Referer: xxx\r\nUser-Agent: xxx\r\n"
    bool isDash;
    int width, height;
};

class StreamExtractor {
public:
    // 检测是否需要提取（非直链 URL）
    static bool needsExtraction(const std::string& url);

    // 同步提取，在后台线程调用，超时 30s
    static bool extract(const std::string& pageUrl,
                        StreamInfo& out,
                        std::string& error);

    // 检测 yt-dlp 是否已安装
    static bool isYtDlpAvailable();
};
```

内部实现：
```bash
yt-dlp -j --no-playlist --no-warnings <url>
```
解析 JSON，优先取合并格式（`vcodec != none && acodec != none`），否则分别取最佳视频流和音频流。

**needsExtraction 判断逻辑：**
- 不含媒体扩展名（`.mp4/.m3u8/.flv` 等）
- 不以 `rtsp://` / `rtmp://` 开头
- 域名在已知平台列表中（bilibili.com、youtube.com、youtu.be、douyin.com、iqiyi.com、youku.com 等）
- 或：URL path 不含文件扩展名且域名非 CDN

### 2. DashMerger（新增）

**文件：**
- `include/FluxPlayer/utils/DashMerger.h`
- `src/utils/DashMerger.cpp`

负责启动 ffmpeg 子进程，将分离的视频/音频流合并为单一 MKV 流，通过匿名管道输出：

```cpp
class DashMerger {
public:
    // 启动合并进程，返回可读 fd（POSIX）或 HANDLE（Windows）
    // startSeconds > 0 时通过 FFmpeg "ss" 选项让上游用 HTTP Range 跳过前面数据
    // （DASH 流 seek 时复用此参数重启 merger，绕开 pipe 不可 seek 的限制）
    bool start(const std::string& videoUrl,
               const std::string& audioUrl,
               const std::string& headers,
               double startSeconds = 0.0);

    // 返回 Demuxer 可打开的 URL（"pipe:N" 或命名管道路径）
    std::string getPipeUrl() const;

    void stop();
    ~DashMerger();

private:
    pid_t ffmpegPid_ = -1;
    int pipeFd_ = -1;
};
```

ffmpeg 命令：
```bash
ffmpeg -loglevel quiet \
  -headers "<headers>" -i <videoUrl> \
  -headers "<headers>" -i <audioUrl> \
  -c copy -f matroska pipe:1
```

**DASH 流 seek 实现**：pipe 输入对 FFmpeg 不可 seek，matroska 流式生成时
也没有 cues 索引，传统 `demuxer_->seek` 必然失败（"Seek to desired resync
point failed"）。`Player::restartDashMerger(seekTime)` 通过停止旧 merger、
重启上游连接来实现真正的 seek：在两个 input 打开后调用
`av_seek_frame(ctx, -1, ts, AVSEEK_FLAG_BACKWARD)`，FFmpeg 内部利用源 m4s URL
的 HTTP Range 让服务端从目标位置开始返回数据。**输出 PTS 保留原始绝对时间**，
Player 时钟无需偏移修正。`processSeekRequest` 检测 `dashMerger_ != nullptr`
时分流到此路径，本地文件 / 普通直链仍走原 `demuxer_->seek`。

**为什么不用 `-ss` 选项**：`av_dict_set(opts, "ss", ...)` 在 `avformat_open_input`
之前注入等价于 ffmpeg CLI 的"input 前 -ss"，会重置输出 PTS 从 0 开始，
导致 AClock=132.8 但实际帧 PTS=0 → 进度条与画面错乱。必须用 input 打开后的
`av_seek_frame` 才能保留原 PTS。

### 4. Player 改造

`Player::open()` 增加提取流程：

```cpp
bool Player::open(const std::string& path) {
    std::string actualUrl = path;
    std::string headers;

    if (StreamExtractor::needsExtraction(path)) {
        setState(EXTRACTING);  // 新增状态，UI 显示"正在解析..."
        StreamInfo info;
        std::string error;
        if (!StreamExtractor::extract(path, info, error)) {
            setError(error);
            return false;
        }
        title_ = info.title;
        if (info.isDash) {
            dashMerger_ = std::make_unique<DashMerger>();
            dashMerger_->start(info.videoUrl, info.audioUrl, info.headers);
            actualUrl = dashMerger_->getPipeUrl();
        } else {
            actualUrl = info.videoUrl;
            headers = info.headers;
        }
    }

    // 原有流程
    demuxer_->open(actualUrl, headers);
    ...
}
```

## Seek 与倍速支持

### 现有实现
- `Player::seek(double seconds)`：非直播流正常 seek，直播流跳过
- `Player::setPlaybackSpeed(double speed)`：通过 `playbackRate_` 控制，音频用 swresample 变速，视频丢帧追赶
- `isLiveStream()` 判断依据：`duration == AV_NOPTS_VALUE` 或 `<= 0`

### DASH pipe 流的问题
通过 `pipe:` 打开的合并流，FFmpeg 无法从容器头读取 duration，导致 `isLiveStream()` 返回 true，seek 和倍速被禁用。

### 解决方案
yt-dlp JSON 中包含 `duration` 字段（单位：秒），在 `DashMerger` 启动后、`Demuxer::open()` 返回前，手动注入 duration：

```cpp
// Demuxer::open() 中，avformat_open_input 成功后
if (knownDuration > 0) {
    m_formatCtx->duration = static_cast<int64_t>(knownDuration * AV_TIME_BASE);
}
```

`StreamInfo` 新增 `duration` 字段，由 `StreamExtractor` 从 yt-dlp JSON 读取并传递给 `DashMerger` / `Demuxer`。

这样 DASH 流的 seek 和倍速行为与本地文件完全一致。

## Cookie 支持（会员内容）

### 原理

平台通过 cookie 中的登录态（如 B站的 `SESSDATA`）判断会员权限。yt-dlp 可直接读取本机浏览器 cookie 数据库，提取时自动携带，平台认为是已登录会员在请求。

### 默认浏览器检测

`StreamExtractor` 启动时自动检测系统默认浏览器，无需用户手动配置：

```cpp
// StreamExtractor.cpp
static std::string detectDefaultBrowser() {
#ifdef __APPLE__
    // 读取 macOS 默认 http handler
    // 调用: defaults read com.apple.LaunchServices/com.apple.launchservices.secure LSHandlers
    // 或使用 LSCopyDefaultHandlerForURLScheme("http") — 需要 CoreServices.framework
    FILE* p = popen("defaults read ~/Library/Preferences/com.apple.LaunchServices/"
                    "com.apple.launchservices.secure 2>/dev/null"
                    " | grep -A1 'LSHandlerURLScheme = http;'"
                    " | grep LSHandlerRoleAll | awk -F'\"' '{print $2}'", "r");
    // 映射 bundle id → yt-dlp browser name
    // com.google.chrome → chrome
    // com.apple.safari  → safari
    // org.mozilla.firefox → firefox
    // com.microsoft.edgemac → edge
#elif defined(_WIN32)
    // 读取注册表 HKCU\Software\Microsoft\Windows\Shell\Associations\UrlAssociations\http\UserChoice
#endif
}
```

优先级：`cookies_browser` 配置项（用户手动指定）> 自动检测默认浏览器 > `cookies_file`。

### 配置项（fluxplayer.ini）

```ini
[Stream]
# 说明：从指定浏览器读取 cookies，用于播放需要登录的内容（如会员视频）
# 取值：chrome / safari / firefox / edge / auto（自动检测默认浏览器）/ off（禁用）
# 默认：auto
cookies_browser = auto

# 说明：cookies.txt 文件路径（Netscape 格式），cookies_browser = off 时生效
# 取值：绝对路径，留空则不使用
# 默认：（空）
cookies_file =
```

遵守守则第七条，`Config::load()` 中同步补充热加载逻辑。

### 平台注意事项

- macOS Chrome/Edge：读取时触发系统钥匙串授权弹窗（首次）
- macOS Safari：yt-dlp 支持良好，无额外授权
- 需用户在默认浏览器中已登录目标平台

### Windows Cookie 文件锁回退

Windows 上 Chromium 系浏览器（Chrome/Edge）即使关闭窗口，后台进程仍会锁住 cookie 数据库文件（SQLite），导致 yt-dlp 内部的 `shutil.copy2()` 失败。

`StreamExtractor::prepareCookieArg()` 实现了三级降级策略：

```
1. 复制 cookie 数据库到缓存目录（仅 Windows + Chrome/Edge）
   ├── 用 CreateFileA + FILE_SHARE_READ|WRITE|DELETE 绕过文件锁
   ├── 复制 Cookies 数据库 + Local State 解密密钥
   ├── 缓存目录与 fluxplayer.ini 同级：%LOCALAPPDATA%\FluxPlayer\browser_cookies\
   └── 成功 → --cookies-from-browser edge:"<缓存profile路径>"

2. 直接读取（复制失败时回退，或 Firefox/macOS/Linux）
   └── --cookies-from-browser <browser>

3. 无 cookie（extract() 中检测到失败后自动降级）
   └── 不带 cookie 参数重试
```

缓存目录结构：
```
%LOCALAPPDATA%\FluxPlayer\
├── fluxplayer.ini
├── browser_cookies\
│   └── edge\                       ← 按默认浏览器名分目录
│       ├── Local State             ← AES 解密密钥
│       └── Default\Network\Cookies ← cookie 数据库副本
```

卸载时删除 `%LOCALAPPDATA%\FluxPlayer` 即可全部清理。

### 手动导出 Cookie 文件（备选方案）

当自动读取 cookie 失败时（浏览器后台进程锁住数据库），可手动导出 Netscape 格式的 cookies.txt 文件，在配置中设置 `cookiesBrowser = off` 并指定 `cookiesFile` 路径。

#### 方法一：浏览器扩展导出（推荐）

在浏览器扩展商店安装以下任一扩展，导出 Netscape 格式的 cookies.txt 文件：

| 扩展名称 | 安装地址 |
|---------|---------|
| Get cookies.txt LOCALLY | [Chrome Web Store](https://chromewebstore.google.com/detail/get-cookiestxt-locally/cclelndahbckbenkjhflpdbgdldlbecc) |
| Cookie.txt Export — LOCALLY | [Chrome Web Store](https://chromewebstore.google.com/detail/cookietxt-export-%E2%80%94-locall/colenanmocikpcfnjgpdaglneaceamfc) |
| cookies.txt | [Edge Add-ons](https://microsoftedge.microsoft.com/addons/detail/ebbdheafhjncoeidpdijmfmkicnejelp) |
| cookies.txt (另一款) | [Edge Add-ons](https://microsoftedge.microsoft.com/addons/detail/cookies-txt/dilbcaaegopfblcjdjikanigjbcbngbk) |
| EditThisCookie (V3) | [Chrome Web Store](https://chromewebstore.google.com/detail/editthiscookie-v3/ojfebgpkimhlhcblbalbfjblapadhbol) |

> Chrome Web Store 的扩展也可以在 Edge 中安装（Edge 设置 → 扩展 → 允许来自其他商店的扩展）。扩展名称中带 **LOCALLY** 的版本数据仅在本地处理，不会上传到服务器，更安全。

导出步骤：
1. 安装上述任一扩展
2. 打开目标网站（如 bilibili.com），确保已登录
3. 点击扩展图标 → 导出 / Export → 保存为 `cookies.txt`
4. 在 `fluxplayer.ini` 中配置：
   ```ini
   cookiesBrowser = off
   cookiesFile = D:/cookies.txt
   ```

> 注意：cookie 有有效期，过期后需重新导出。不要将 cookie 文件分享给他人，等同于账号密码。

#### 方法二：直接复制浏览器 Cookie 数据库

关闭浏览器所有窗口和后台进程后，可直接复制数据库文件。各浏览器 cookie 数据库路径如下：

| 浏览器 | Cookie 数据库路径 |
|--------|------------------|
| Google Chrome | `C:\Users\<用户名>\AppData\Local\Google\Chrome\User Data\Default\Network\Cookies` |
| Microsoft Edge | `C:\Users\<用户名>\AppData\Local\Microsoft\Edge\User Data\Default\Network\Cookies` |
| Firefox | `C:\Users\<用户名>\AppData\Roaming\Mozilla\Firefox\Profiles\<profile>.default-release\cookies.sqlite` |
| 360 浏览器 | `C:\Users\<用户名>\AppData\Roaming\360se6\User Data\Default\Network\Cookies` |
| macOS Chrome | `~/Library/Application Support/Google/Chrome/Default/Network/Cookies` |
| macOS Edge | `~/Library/Application Support/Microsoft Edge/Default/Network/Cookies` |
| macOS Safari | 由系统 Keychain 管理，yt-dlp 直接支持 `--cookies-from-browser safari` |

操作步骤：
1. 完全关闭浏览器（包括后台进程，可在任务管理器中确认无 `msedge.exe` / `chrome.exe`）
2. 复制上表中的 Cookies 文件到任意位置
3. FluxPlayer 的自动复制机制会在浏览器关闭时自动成功，无需手动操作

> Edge 默认开启"启动增强"，关闭窗口后仍有后台进程。关闭方法：Edge 设置 → 系统和性能 → 关闭"启动增强"和"在 Microsoft Edge 关闭后继续运行后台扩展和应用"。

## 画质选择

### 方案

yt-dlp JSON 中 `formats` 数组包含所有可用画质，每条记录含 `format_id`、`height`、`vcodec`、`filesize` 等字段。

### 流程

```
StreamExtractor::extract() 时同时提取所有可用画质列表
    │
    ▼
Player 开始播放后，Controller 工具栏显示当前画质（如 "1080P"）
    │
    ▼
用户点击画质按钮 → 弹出画质菜单（360P / 480P / 720P / 1080P / 4K）
    │
    ▼
选择新画质 → 记录当前播放位置 currentTime
    │
    ▼
重新调用 StreamExtractor::extract(pageUrl, selectedFormatId)
    │
    ▼
Player::open(newUrl) + seek(currentTime)  ← 无缝切换，保持播放位置
```

### StreamInfo 扩展

```cpp
struct QualityOption {
    std::string formatId;   // yt-dlp format_id，传给 -f 参数
    int height;             // 360 / 480 / 720 / 1080 / 2160
    std::string label;      // "1080P" / "4K" 等显示名
    int64_t filesize;       // 字节，-1 表示未知
};

struct StreamInfo {
    // ... 原有字段 ...
    std::vector<QualityOption> qualities;  // 新增：所有可用画质
    std::string selectedFormatId;          // 当前选中的 format_id
};
```

### yt-dlp 调用

提取时加 `-f <formatId>` 指定画质：

```cpp
// 首次提取：不指定 format，取默认最佳画质并返回所有可用画质列表
// 切换画质：指定 formatId，重新提取对应流 URL
cmd.push_back("-f");
cmd.push_back(formatId.empty() ? "bestvideo+bestaudio/best" : formatId);
```

## 下载功能

### 方案

yt-dlp 已提取到 `StreamInfo`，下载只需把流写入本地文件。直接调用 yt-dlp 下载比自己实现更可靠（自动处理 DASH 合并、断点续传、格式转换）：

```bash
yt-dlp -f "bestvideo+bestaudio" --merge-output-format mp4 \
  --cookies-from-browser <browser> \
  -o "<outputDir>/%(title)s.%(ext)s" \
  <pageUrl>
```

### 新增 Downloader 类

**文件：** `include/FluxPlayer/utils/Downloader.h` / `src/utils/Downloader.cpp`

```cpp
class Downloader {
public:
    struct Task {
        std::string pageUrl;
        std::string outputDir;
        std::function<void(float progress)> onProgress;  // 0.0 ~ 1.0
        std::function<void(bool ok, std::string path)>  onFinish;
    };

    // 异步下载，立即返回，回调在主线程触发
    void start(const Task& task);
    void cancel();

private:
    pid_t pid_ = -1;
};
```

内部解析 yt-dlp 的 stderr 输出（`[download] 45.3% ...`）提取进度，通过 `onProgress` 回调更新 UI。

### UI 入口

播放器 Controller 工具栏新增"下载"按钮，仅当 `lastPageUrl_` 非空（即当前播放内容来自网页 URL）时显示。点击后弹出系统文件夹选择对话框，确认后启动下载，底部状态栏显示进度条。
