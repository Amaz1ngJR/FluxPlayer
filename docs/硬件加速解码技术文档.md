## 为什么需要硬件解码

### 软件解码的瓶颈

视频解码是计算密集型任务。以 4K H.265 视频为例：

```
4K 分辨率 = 3840 × 2160 = 829 万像素/帧
30fps = 每秒 30 帧
每秒需要处理 ≈ 2.5 亿像素的解码运算
```

纯 CPU 软件解码时，这些运算全部由 CPU 通用计算单元完成，会导致：
- CPU 占用率飙升（4K H.265 可达 80%+）
- 笔记本发热严重、风扇狂转
- 其他应用响应变慢
- 移动设备电池快速消耗

### 硬件解码的优势

现代 GPU 内置了专用的视频解码引擎（固定功能硬件），专门为视频解码设计：

```
CPU 软件解码：通用计算单元 → 灵活但效率低
GPU 硬件解码：专用解码引擎 → 功能固定但效率极高
```

硬件解码的好处：
- **CPU 占用率大幅降低**（从 80% 降到 5% 以下）
- **功耗更低**（专用硬件比通用计算省电得多）
- **解码速度更快**（专用电路的吞吐量远超 CPU）

## CPU 解码 vs GPU 解码

### 数据流对比

```
【CPU 软件解码】
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 压缩数据  │ →  │ CPU 解码  │ →  │ CPU 转换  │ →  │ GPU 渲染  │
│ (H.264)  │    │ (libavcodec)│   │(sws_scale)│    │ (OpenGL) │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                  ↑ CPU 密集      ↑ CPU 密集      ↑ GPU
                  占用高           额外拷贝         轻松

【GPU 硬件解码】
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 压缩数据  │ →  │ GPU 解码  │ →  │ GPU→CPU  │ →  │ GPU 渲染  │
│ (H.264)  │    │(VideoToolbox)│  │ (传输)   │    │ (OpenGL) │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                  ↑ GPU 专用       ↑ 一次拷贝       ↑ GPU
                  CPU 几乎空闲     不可避免          轻松
```

### 关键区别

| 维度 | CPU 软件解码 | GPU 硬件解码 |
|------|-------------|-------------|
| 解码执行者 | CPU 通用计算单元 | GPU 专用解码引擎 |
| CPU 占用 | 高（50-100%） | 极低（<5%） |
| 输出格式 | 通常 YUV420P | 通常 NV12 |
| 兼容性 | 支持所有编解码器 | 仅支持常见编解码器 |
| 额外开销 | 无 | GPU→CPU 帧传输 |

## 各平台硬件解码方案

### macOS: VideoToolbox

```
Apple 的官方多媒体框架，macOS / iOS 通用
├── 支持编解码器：H.264, HEVC(H.265), VP9, ProRes
├── FFmpeg 设备类型：AV_HWDEVICE_TYPE_VIDEOTOOLBOX
├── 输出像素格式：AV_PIX_FMT_VIDEOTOOLBOX → 传输后 NV12
├── 底层实现：Apple Silicon 的专用媒体引擎 / Intel Mac 的 Quick Sync
└─�� 特点：macOS 上唯一的硬件解码方案，无替代品
```

### Windows: 三种方案

```
1. CUDA / NVDEC（NVIDIA 专用）
   ├── 仅 NVIDIA GPU 可用
   ├── FFmpeg 设备类型：AV_HWDEVICE_TYPE_CUDA
   ├── 输出像素格式：AV_PIX_FMT_CUDA → 传输后 NV12
   ├── 支持：H.264, HEVC, VP9, AV1（RTX 30+）
   └── 特点：性能最优，NVIDIA 专用解码芯片 NVDEC

2. D3D11VA（通用方案）
   ├── 所有支持 DirectX 11 的 GPU（Intel/AMD/NVIDIA）
   ├── FFmpeg 设备类型：AV_HWDEVICE_TYPE_D3D11VA
   ├── 输出像素格式：AV_PIX_FMT_D3D11 → 传输后 NV12
   ├── 支持：H.264, HEVC, VP9（取决于 GPU 驱动）
   └── 特点：Win8+ 通用，覆盖面最广

3. DXVA2（旧版回退）
   ├── 所有支持 DirectX 9 的 GPU
   ├── FFmpeg 设备类型：AV_HWDEVICE_TYPE_DXVA2
   ├── 输出像素格式：AV_PIX_FMT_DXVA2_VLD → 传输后 NV12
   ├── 支持：H.264, HEVC
   └── 特点：WinXP+ 兼容，作为 D3D11VA 不可用时的回退
```

### 优先级选择策略

```
macOS:   VideoToolbox（唯一选择）
Windows: CUDA(NVDEC) → D3D11VA → DXVA2（按优先级依次尝试）
Linux:   暂不支持（可扩展 VAAPI）
```

FluxPlayer 的实现方式是定义一个候选列表，循环尝试 `av_hwdevice_ctx_create()`，
第一个成功的就用。这种设计的好处是：
- 新增平台只需往列表里加一项
- 自动适配用户的 GPU 硬件
- 失败自动降级，无需用户干预

## FFmpeg 硬件加速 API 详解

### 核心 API 调用流程

```
┌─────────────────────────────────────────────────────────────┐
│                    FFmpeg 硬件加速流程                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. av_hwdevice_ctx_create()     创建硬件设备上下文           │
│         ↓                                                   │
│  2. codecCtx->hw_device_ctx =    绑定到解码器上下文           │
│         av_buffer_ref(hwDeviceCtx)                          │
│         ↓                                                   │
│  3. avcodec_open2()              打开解码器（FFmpeg 自动       │
│         ↓                        协商硬件格式）               │
│  4. avcodec_send_packet()        发送压缩数据（与软件解码相同） │
│         ↓                                                   │
│  5. avcodec_receive_frame()      接收解码帧                   │
│         ↓                        ⚠ 帧在 GPU 内存中！         │
│  6. av_hwframe_transfer_data()   GPU → CPU 帧传输            │
│         ↓                        输出 NV12 格式              │
│  7. 渲染 NV12 帧                                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1. av_hwdevice_ctx_create() — 创建硬件设备

```c
int av_hwdevice_ctx_create(
    AVBufferRef **device_ctx,    // [输出] 硬件设备上下文
    enum AVHWDeviceType type,    // 设备类型（如 AV_HWDEVICE_TYPE_VIDEOTOOLBOX）
    const char *device,          // 设备标识（通常传 NULL，使用默认设备）
    AVDictionary *opts,          // 额外选项（通常传 NULL）
    int flags                    // 标志位（通常传 0）
);
// 返回值：0 = 成功，负数 = 失败
```

这个函数做了什么：
- 检测系统是否有对应的硬件解码能力
- 初始化硬件设备（打开 GPU 驱动、分配设备资源）
- 返回一个 `AVBufferRef*`，内部包含设备句柄

如果系统没有对应的 GPU 或驱动不支持，这个函数会返回错误码，
此时我们可以尝试下一个候选方案或回退到软件解码。

### 2. hw_device_ctx 绑定 — 告诉解码器使用硬件

```c
// 将硬件设备上下文绑定到解码器
codecCtx->hw_device_ctx = av_buffer_ref(hwDeviceCtx);
```

`av_buffer_ref()` 增加引用计数（不是拷贝），这样解码器和我们各持有一个引用。
绑定后，`avcodec_open2()` 会自动检测该设备支持哪些编解码器，
并在可能的情况下使用硬件解码路径。

### 3. av_hwframe_transfer_data() — GPU→CPU 帧传输

```c
int av_hwframe_transfer_data(
    AVFrame *dst,    // [输出] CPU 端帧（格式由 FFmpeg 自动选择，通常 NV12）
    AVFrame *src,    // [输入] GPU 端帧（格式为 AV_PIX_FMT_VIDEOTOOLBOX 等）
    int flags        // 标志位（通常传 0）
);
```

这是硬件解码中**唯一不可避免的拷贝**：将解码结果从 GPU 显存传输到 CPU 内存。
传输后的帧格式通常是 NV12（所有主流硬件解码器的默认输出格式）。

### 引用计数与内存管理

```
AVBufferRef 引用计数机制：

av_hwdevice_ctx_create(&ctx)  →  ctx 引用计数 = 1
av_buffer_ref(ctx)            →  ctx 引用计数 = 2（解码器持有一份）
av_buffer_unref(&ctx)         →  ctx 引用计数 = 1（我们释放，解码器还在用）
avcodec_free_context()        →  ctx 引用计数 = 0（解码器释放，设备真正销毁）
```

## NV12 像素格式深度解析

### 为什么硬件解码器输出 NV12 而不是 YUV420P？

两者都是 YUV 4:2:0 采样（色度分辨率为亮度的 1/4），区别在于**内存布局**：

```
【YUV420P — 平面存储（Planar）】
三个独立的内存平面，U 和 V 分开存储：

Y 平面（全分辨率）：
┌────────────────────┐
│ Y00 Y01 Y02 Y03 ...│  width × height 字节
│ Y10 Y11 Y12 Y13 ...│
│ ...                 │
└────────────────────┘

U 平面（1/4 分辨率）：
┌──────────┐
│ U00 U01 ..│  width/2 × height/2 字节
│ U10 U11 ..│
└──────────┘

V 平面（1/4 分辨率）：
┌──────────┐
│ V00 V01 ..│  width/2 × height/2 字节
│ V10 V11 ..│
└──────────┘

总内存：width × height × 1.5 字节
data[0] = Y, data[1] = U, data[2] = V


【NV12 — 半平面存储（Semi-Planar）】
两个内存平面，U 和 V 交错存储：

Y 平面（全分辨率）：
┌────────────────────┐
│ Y00 Y01 Y02 Y03 ...│  width × height 字节
│ Y10 Y11 Y12 Y13 ...│
│ ...                 │
└────────────────────┘

UV 交错平面（1/2 分辨率）：
┌──────────────────────┐
│ U00 V00 U01 V01 ...  │  width × height/2 字节
│ U10 V10 U11 V11 ...  │  （每个像素 2 字节：U + V）
└──────────────────────┘

总内存：width × height × 1.5 字节（与 YUV420P 相同）
data[0] = Y, data[1] = UV 交错
```

### NV12 对硬件更友好的原因

1. **缓存局部性**：U 和 V 交错存储，访问一个像素的色度信息只需一次内存读取
2. **DMA 效率**：两个平面比三个平面需要更少的 DMA 传输操作
3. **GPU 纹理映射**：NV12 的 UV 平面可以直接映射为 GL_RG8 纹理（双通道）

这就是为什么所有主流硬件解码器（VideoToolbox、NVDEC、D3D11VA）都输出 NV12。

## 零拷贝渲染技术

### 传统方案的问题

```
【有冗余拷贝的流程】
GPU 解码 → av_hwframe_transfer_data → CPU(NV12) → sws_scale → CPU(YUV420P) → glTexSubImage2D → GPU
                                                    ^^^^^^^^
                                                    冗余！NV12 和 YUV420P 信息量完全相同
                                                    sws_scale 做了一次无意义的像素重排
```

`sws_scale` 的 NV12→YUV420P 转换本质上只是把交错的 UV 拆成独立的 U 和 V 平面，
没有任何信息增减，纯粹是内存拷贝 + 重排。对于 1080p 视频，这意味着每帧浪费约 1.5MB 的内存带宽。

### 零拷贝方案

```
【零拷贝流程】
GPU 解码 → av_hwframe_transfer_data → CPU(NV12) → glTexSubImage2D → GPU
                                                    ↑
                                                    直接上传 NV12 数据到 GPU 纹理
                                                    跳过 sws_scale！
```

关键技术：**让 OpenGL 着色器直接处理 NV12 格式**。

### OpenGL NV12 纹理方案

```
YUV420P 需要 3 个纹理：
  纹理 0 (GL_RED):  Y 平面  → width × height
  纹理 1 (GL_RED):  U 平面  → width/2 × height/2
  纹理 2 (GL_RED):  V 平面  → width/2 × height/2

NV12 只需要 2 个纹理：
  纹理 0 (GL_RED):  Y 平面  → width × height
  纹理 1 (GL_RG8):  UV 平面 → width/2 × height/2（双通道：R=U, G=V）
```

`GL_RG8` 是 OpenGL 的双通道纹理格式，每个纹素包含 R 和 G 两个 8 位通道。
NV12 的 UV 交错数据天然匹配这个格式：第一个字节是 U（映射到 R），第二个字节是 V（映射到 G）。

### 着色器实现

```glsl
uniform int isNV12;       // 0 = YUV420P, 1 = NV12
uniform sampler2D texY;   // Y 平面（两种格式共用）
uniform sampler2D texU;   // YUV420P: U平面(R) / NV12: UV平面(RG)
uniform sampler2D texV;   // 仅 YUV420P 使用

void main() {
    float y = texture(texY, TexCoord).r;
    float u, v;

    if (isNV12 == 1) {
        // NV12：从 RG 纹理的两个通道分别读取 U 和 V
        vec2 uv = texture(texU, TexCoord).rg;
        u = uv.r - 0.5;
        v = uv.g - 0.5;
    } else {
        // YUV420P：U 和 V 各自独立的单通道纹理
        u = texture(texU, TexCoord).r - 0.5;
        v = texture(texV, TexCoord).r - 0.5;
    }

    // BT.601 YUV→RGB 转换
    float r = y + 1.402 * v;
    float g = y - 0.344 * u - 0.714 * v;
    float b = y + 1.772 * u;
    FragColor = vec4(r, g, b, 1.0);
}
```

> `isNV12` 是 uniform 变量（所有片段取同一值），GPU 不存在分支发散惩罚，性能影响为零。

### 内存优化汇总

| 优化项 | 技术 | 效果 |
|--------|------|------|
| 跳过 sws_scale | NV12 直接渲染 | 省去一次全帧像素拷贝+重排 |
| 复用传输帧 | `m_hwTransferFrame` 成员 | 避免每帧 `av_frame_alloc`/`av_frame_free` |
| 引用计数入队 | `av_frame_ref` | 仅增加引用计数，不复制像素数据 |
| 格式变化检测 | `m_lastSwsFormat` 守卫 | 避免 SwsContext 格式不匹配导致崩溃 |

## FluxPlayer 实现细节

### 完整数据流

```
【硬件解码路径（默认）】

                    ┌─ av_hwdevice_ctx_create() ─┐
                    │   创建硬件设备上下文          │
                    └─────────────┬───────────────┘
                                  ↓
AVPacket ──→ avcodec_send_packet() ──→ avcodec_receive_frame()
                                              ↓
                                    AVFrame (GPU 内存)
                                    format = AV_PIX_FMT_VIDEOTOOLBOX
                                              ↓
                                    av_hwframe_transfer_data()
                                    GPU → CPU 传输
                                              ↓
                                    AVFrame (CPU 内存)
                                    format = AV_PIX_FMT_NV12
                                              ↓
                                    av_frame_ref() 零拷贝入队
                                              ↓
                                    ┌─────────────────────┐
                                    │   视频帧队列          │
                                    └─────────┬───────────┘
                                              ↓
                                    glTexSubImage2D()
                                    Y → GL_RED 纹理
                                    UV → GL_RG8 纹理
                                              ↓
                                    片段着色器 NV12→RGB
                                              ↓
                                           屏幕


【软件解码路径（回退）】

AVPacket ──→ avcodec_send_packet() ──→ avcodec_receive_frame()
                                              ↓
                                    AVFrame (CPU 内存)
                                    format = AV_PIX_FMT_YUV420P
                                              ↓
                                    av_frame_ref() 零拷贝入队
                                              ↓
                                    glTexSubImage2D()
                                    Y → GL_RED, U → GL_RED, V → GL_RED
                                              ↓
                                    片段着色器 YUV420P→RGB
                                              ↓
                                           屏幕
```

## 降级策略设计

### 多层降级保障

```
层级 1：配置文件控制
  hwaccel=false → 完全跳过硬件初始化，纯软件解码

层级 2：硬件设备初始化失败
  av_hwdevice_ctx_create() 返回错误
  → 尝试下一个候选方案（CUDA → D3D11VA → DXVA2）
  → 所有候选都失败 → 自动回退软件解码
  → 日志输出警告，用户无感知

层级 3：帧传输失败
  av_hwframe_transfer_data() 返回错误
  → prepareFrame() 返回 false
  → Player 丢弃该帧（已有的帧丢弃逻辑）
  → 不影响后续帧的解码

层级 4：编解码器不支持硬件
  例如播放 VP8 格式（硬件不支持）
  → FFmpeg 自动使用软件解码路径
  → 输出 YUV420P 格式
  → prepareFrame() 走 YUV420P 零拷贝路径
  → 完全透明，无需特殊处理
```

### 设计原则

1. **失败不崩溃**：所有硬件相关操作的失败都有明确的回退路径
2. **用户无感知**：降级过程自动完成，仅在日志中记录
3. **最小影响范围**：硬件初始化失败只影响 VideoDecoder，不影响其他模块
4. **可配置**：用户可以通过 `hwaccel=false` 强制使用软件解码（调试用）
